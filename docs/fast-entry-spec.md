# Fast Entry (Inventory) — спецификация UX и инварианты

Цель: быстрый, устойчивый к сети, keyboard-first ввод позиций ревизии. Документ задаёт инварианты UX, которые нельзя «случайно сломать» при рефакторингах.

Область действия: экран ревизии (Fast Entry) и связанные механики сохранения/оффлайна/журнала.

## 1) Keyboard-first flow (основной сценарий)

Пользователь работает без мыши.

### Сценарий: поиск → выбор → qty → Enter → сохранение → фокус на поиск

1. Фокус по умолчанию в поле поиска.
2. Пользователь печатает запрос.
3. Через debounce выполняется поиск (или показываются подсказки).
4. Пользователь выбирает позицию:
   - стрелки ↑/↓ двигают highlight;
   - Enter выбирает highlighted (или первый, если highlight отсутствует);
   - Escape закрывает dropdown.
5. После выбора позиции:
   - dropdown закрывается;
   - фокус переходит в поле количества;
   - рядом отображается unit выбранного товара (display-only).
6. Пользователь вводит количество.
7. Enter в поле количества запускает сохранение.
8. После успешного сохранения:
   - количество сбрасывается;
   - выбранный товар сбрасывается;
   - при `clearSearchAfterSave=true` поле поиска очищается;
   - фокус возвращается в поле поиска.

Инвариант: после успешного сохранения всегда можно сразу начать ввод следующей позиции с клавиатуры.

## 2) Поиск: правила и параметры

### 2.1 Debounce и лимит

- Debounce: 150 мс (целевое значение для UX; допустимо 100–250 мс).
- Лимит результатов: 20.

### 2.2 Порядок источников

- Если строка поиска непуста → backend search (`/items/search?q=...`).
- Если строка поиска пуста → показывать «чипы» подсказок:
  1. Избранное
  2. Частые
  3. Недавние

### 2.3 Aliases

- Поиск должен поддерживать алиасы (синонимы) на стороне бэкенда.
- UX-правило: пользователь не должен знать «каноническое имя». Достаточно ввести распространённый синоним.

Примечание: алиасы — часть каталога, а не логика клиента.

## 3) Autosave / Persistence

Два слоя сохранения состояния:

### 3.1 Draft (черновик) в localStorage

Черновик предназначен для восстановления незавершённого ввода (перезагрузка страницы, краш вкладки, случайный переход).

- Хранилище: `localStorage`
- Ключ: версионированный (`inventory-fast-entry-draft-v1`).
- Скоуп черновика: минимум `username + warehouse + zone`.
- Содержимое черновика:
  - `searchTerm` (string)
  - `selectedItem` (минимальный набор полей для восстановления выбора)
  - `qty` (string)
  - `updatedAt` (epoch ms)
- Восстановление:
  - только если текущие поля пустые (не перетирать активный ввод);
  - если черновик старше 7 дней → удалить.

### 3.2 Offline queue в IndexedDB (с fallback)

Очередь предназначена для гарантии доставки при нестабильной сети.

- Primary storage: IndexedDB
- Fallback: localStorage
- Ключ: `inventory_pending_entries_v1` (текущая реализация)
- Элемент очереди включает:
  - `idempotency_key`
  - `session_id`, `warehouse_id`
  - `item_id`, `item_name`, `unit`
  - `qty`, `mode`
  - `created_at`
  - `retry_count`, `next_retry_at`, `status`

### 3.3 Правило очистки черновика

Инвариант: draft очищается **только после успешного add**:

- либо успешного сохранения на сервер,
- либо успешного добавления в offline queue.

Нельзя очищать draft «просто потому что нажали Enter» или «просто потому что запрос начался».

## 4) Last entries (журнал последних действий)

### 4.1 Что показываем

Журнал объединяет:

- сохранённые на сервере записи (recent entries),
- и локальные pending/failed элементы очереди.

Для строки в журнале показываем:

- статус (`saved`/`pending`/`failed`)
- название товара
- qty + unit
- timestamp
- (если доступно) кто внёс (actor)
- (если доступно) counted outside zone + countedByZone

### 4.2 Порядок

- Сортировка по timestamp по убыванию.

### 4.3 Защита от дублей

Инварианты:

- один offline queue item идентифицируется по `idempotency_key`;
- одна серверная запись идентифицируется по `entry.id`;
- при успешной синхронизации offline queue item должен исчезнуть из очереди (и, следовательно, из pending-части журнала), чтобы не оставаться дублем относительно server recent.

Допустимо кратковременное «двойное присутствие», если UI показывает pending, а сервер уже принял, но очередь ещё не обновилась. При следующем цикле sync это должно исчезнуть.

## 5) Units (единицы измерения)

- Unit (`uom`) — **display-only**.
- Никаких конверсий (kg↔g, l↔ml, pcs↔box) на клиенте.
- Подсказка упаковки (если нужна) — только текстом (например: "1 короб = 12 pcs"), без автоматических пересчётов.

## 6) Offline / Online

### 6.1 Поведение при offline

- При попытке сохранить позицию в offline:
  - добавляем элемент в offline queue;
  - показываем пользователю явный статус ("В очереди" / "Нет сети — сохранили в очередь");
  - UI сбрасывает ввод так же, как при обычном успехе (чтобы сохранить keyboard-first поток).

### 6.2 Автосинк

- При переходе в online запускается sync.
- Пока очередь непуста — периодический sync (например, каждые ~12 секунд).
- Retry policy:
  - экспоненциальная задержка;
  - верхняя граница задержки.

### 6.3 Ошибки синка

- 409 (идемпотентность/конфликт) не должен приводить к бесконечным ретраям; запись помечается как конфликтная и требует внимания.
- сетевые ошибки → перевод в failed + scheduled retry.

## 7) Conflict policy (HTTP 409)

Ключевой принцип: 409 — это сигнал «состояние изменилось, повторите ввод осознанно».

Политика для Fast Entry:

- при 409 во время сохранения:
  - показать пользователю чёткое сообщение "Конфликт: обновлено другим действием. Повторите ввод";
  - обновить данные, влияющие на ввод/журнал (invalidate recent/audit/progress);
  - не терять набранное количество без необходимости.

Примечание: точная стратегия может зависеть от причины 409 (версии/идемпотентность). UI должен вести себя предсказуемо.

## 8) Acceptance criteria (15–20)

1. Фокус при открытии страницы ревизии находится в поле поиска.
2. Ввод текста в поиск запускает backend search после debounce.
3. Поиск не выполняется, если строка после trim пуста.
4. Максимум 20 результатов в dropdown.
5. ArrowDown/ArrowUp меняют highlighted элемент в dropdown.
6. Enter в поле поиска выбирает highlighted (или первый элемент) и переводит фокус в qty.
7. Escape закрывает dropdown и сбрасывает highlight.
8. Qty input недоступен, пока не выбран товар.
9. Enter в qty запускает сохранение.
10. При успешном сохранении на сервере: qty очищается, selectedItem очищается, фокус возвращается в поиск.
11. При offline сохранение добавляет элемент в очередь и показывает статус "В очереди".
12. Draft сохраняется в localStorage при изменении search/selectedItem/qty и восстанавливается после reload.
13. Draft не перетирает активный ввод (если поля уже заполнены).
14. Draft очищается только после успешного add: либо успешной записи на сервер, либо успешной записи в очередь.
15. Offline queue хранится в IndexedDB (с fallback в localStorage) и сохраняется между перезапусками вкладки.
16. При переходе в online синхронизация запускается автоматически.
17. После успешной синхронизации соответствующие элементы исчезают из pending части.
18. Журнал показывает элементы в порядке убывания времени.
19. При 409 во время save UI показывает понятное сообщение и просит повторить ввод.
20. Единицы измерения отображаются как текст и не конвертируются.
